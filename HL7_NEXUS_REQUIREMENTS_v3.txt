================================================================================
  SECTION 0 // EXECUTIVE SUMMARY & CORE ARCHITECTURE CONCEPT
================================================================================

HL7_NEXUS is a corporate command-grade HL7 message intelligence platform
built around one defining architectural idea: the ENGINE_MODE system.
Every single operation — generation, parsing, validation, auto-fix — routes
through one of three selectable processing engines. The user picks the engine.
The entire UI reconfigures around that choice.

  ┌─────────────────────────────────────────────────────────────────────┐
  │                       ENGINE_MODE SELECTOR                           │
  │                                                                       │
  │   ┌────────────────────────────────────┐   ┌───────────────────────┐│
  │   │         [  AI ENGINE MODE  ]        │   │  [ ALGORITHM MODE ]  ││
  │   │                                    │   │                       ││
  │   │  Sub-mode A: ☁ CLOUD AI            │   │  Pure deterministic   ││
  │   │               Claude API (Anthropic)│   │  code. No AI calls.   ││
  │   │                                    │   │  No internet needed.  ││
  │   │  Sub-mode B: ⬡ LOCAL AI            │   │  Rule engine +        ││
  │   │               Ollama (localhost)    │   │  FSM + PEG parser.    ││
  │   │               Any compatible model │   │                       ││
  │   └────────────────────────────────────┘   └───────────────────────┘│
  └─────────────────────────────────────────────────────────────────────┘

This tells judges:
  - The team understands real enterprise constraints (offline, air-gapped,
    cost-control environments don't always have cloud AI)
  - Complex engineering exists at every layer, not just the AI wrapper
  - The system degrades gracefully and is production-honest

WHAT CHANGES PER ENGINE MODE:
  - Left panel agent monitors swap out entirely (AI agents vs algo processors)
  - Right panel intelligence monitor adapts (AI metrics vs algo metrics)
  - Center panel tabs enable/disable context-sensitively (NL_INPUT tab is
    only active in AI mode; it is greyed and locked in Algorithm mode)
  - Event bus terminal shows engine-specific event types
  - Processing pipeline in the backend routes to a completely different
    code path
  - The ENGINE_MODE badge in the global header changes color and label

================================================================================
  SECTION 1 // UI DESIGN SYSTEM — IRON COUNCIL THEME (CORPORATE WHITE)
================================================================================

This is the most important section for the implementing agent. The UI must
match the aesthetic described here precisely. The reference image shows a
corporate command interface — not a dark terminal. It is WHITE and LIGHT,
with deep red structural accents, dark slate headers, and brutalist edges.
Do not use a dark theme. The "system terminal" feeling comes from sharp
edges, dense micro-typography, and data-richness — not from dark backgrounds.

--------------------------------------------------------------------------------
1.1 LAYOUT — THREE FLOATING COLUMN ARCHITECTURE
--------------------------------------------------------------------------------

The root layout is always three vertical columns floating over a light
backdrop. Think of it as three physical hardware panels sitting on a desk.

BACKDROP:
  Background color   : Light cool gray with a slight lavender tint
  CSS value          : #E0E7FF or Tailwind bg-slate-100 / bg-indigo-50
  CRT Scanlines      : A CSS pseudo-element (::after or ::before) overlaid
                       on the entire app background renders horizontal
                       scanlines using repeating-linear-gradient. These are
                       very subtle — low opacity (3–5%) — purely aesthetic.
                       They must not interfere with readability.
                       Implementation:
                         background: repeating-linear-gradient(
                           to bottom,
                           transparent 0px,
                           transparent 3px,
                           rgba(0,0,0,0.03) 3px,
                           rgba(0,0,0,0.03) 4px
                         );
                         pointer-events: none;
                         position: fixed;
                         inset: 0;
                         z-index: 9999;

COLUMN STRUCTURE:
  Left Column        : OPERATIONAL_MONITOR — fixed width ~280px
  Center Column      : COMMAND_INTERFACE — flex-grow, takes remaining space
  Right Column       : INTELLIGENCE_MONITOR — fixed width ~300px
  Column gap         : 0px (columns sit flush against each other)
  Column shadows     : Each column has shadow-xl or shadow-2xl
                       (soft diffused drop shadow — NOT a hard offset shadow)
  Column backgrounds : bg-white (main column bodies)
  Column borders     : Each column has a thin outer border: border border-gray-200

GLOBAL HEADER (above all three columns, full width):
  The global app-level header sits above the three-column layout.
  Background  : bg-white
  Border-bottom: 2px solid #af0a0f (thick red structural divider)
  Contents    :
    - Left: App logo text "HL7_NEXUS" in deep red, font-mono, tracking-widest
    - Center: ENGINE_MODE selector (described in Section 3)
    - Right: Connection status LED + label + session timer

      Role         : Template-based HL7 message generator
      What it does : Takes a JSON param object, selects a pre-built
                     template, performs field interpolation, escapes
                     special characters, applies delimiter encoding
      Metrics shown: template loaded, fields filled, bytes generated
      States       : IDLE / GENERATING / COMPLETE / ERROR

--------------------------------------------------------------------------------
7.3 ALGORITHM MODE (Pure Deterministic — Zero AI)
--------------------------------------------------------------------------------
This is the mode that shows genuine engineering depth. Every operation
that the AI would do in AI mode is replicated by a deterministic algorithm.
No LLM calls, no API keys, no internet required.

HL7 LEXER (LEXER_PROC):
  Input: raw HL7 string
  Algorithm:
    Step 1: Read first 3 characters — must be "MSH"
    Step 2: Read character at index 3 — this is FIELD_SEPARATOR (usually |)
    Step 3: Read characters 4-7 — these are ENCODING_CHARS (^~\&)
    Step 4: Split the entire message by CARRIAGE_RETURN (\r or \r\n)
            to get segment strings
    Step 5: For each segment string, split by FIELD_SEPARATOR to get fields
    Step 6: For each field, split by COMPONENT_SEPARATOR (^) to get components
    Step 7: For each component, split by SUBCOMPONENT_SEPARATOR (&)
    Step 8: Handle escape sequences: \F\ \S\ \E\ \R\ \T\ \H\ \N\
    Output: A nested token tree (Segment[] → Field[] → Component[] → Sub[])
  Error conditions:
    Missing MSH: throw LexerError("Message must begin with MSH segment")
    No field separator at position 3: throw LexerError("Invalid MSH.1")
    Truncated message: throw LexerError("Unexpected end of message")

HL7 PEG PARSER (PARSER_PROC):
  Input: token tree from Lexer
  Algorithm:
    A Parsing Expression Grammar (PEG) for each supported message type.
    The grammar is encoded as a JavaScript/Python object tree:
      Grammar = {
        'ADT_A01': ['MSH', 'EVN', 'PID', 'PD1?', 'NK1*', 'PV1', 'PV2?', ...],
        'ORU_R01': ['MSH', 'PATIENT_RESULT+', 'DSC?'],
        'ORM_O01': ['MSH', 'NTE*', 'PATIENT?', 'ORDER+'],
        ...
      }
    '?' = optional (0 or 1)
    '*' = zero or more
    '+' = one or more
    Groups in brackets = logical groups (like the PATIENT_RESULT group in ORU)
    Parser walks the token tree against the grammar:
      If a required segment is missing: ParserError with segment name
      If segments are out of order: ParserError with expected vs found
      If an unknown segment (Z-segment): flag as warning, not error
    Output: Abstract Syntax Tree (AST) with full message structure
    Also extracts: message type, version, sending app, timestamp

HL7 RULE ENGINE VALIDATOR (VALIDATOR_PROC):
  Input: AST from Parser + selected conformance profile
  Algorithm:
    A compiled rule set. Each rule is a function:
      Rule = { id, severity, segment, field, check_fn, error_message }
    Rules are organized into categories:

    CATEGORY A — Required Field Rules (automatically generated from spec):
      For each segment in HL7 spec: enumerate required fields (R) vs optional (O)
      Rule: if field is R and value is empty → ERROR E001
      Pre-compiled into lookup table: REQUIRED_FIELDS[segment][field_index]

    CATEGORY B — Data Type Rules:
      DT  : must match /^\d{8}$/ (YYYYMMDD) or be empty
      DTM : must match /^\d{8,14}$/ (YYYYMMDDHHMMSS with variable length)
      TM  : must match /^\d{4,10}(\.\d{4})?([+-]\d{4})?$/
      NM  : must match /^-?\d+(\.\d+)?$/
      SI  : must match /^\d+$/
      IS  : value must be in specified HL7 table (loaded from bundled tables)
      CWE : three-component structure: code^display^codeSystem
      CE  : similar to CWE (older version)
      ID  : value must be in specified HL7 table
      ST  : any string, max length check
      TX  : any text, max length per spec

    CATEGORY C — Table Validation Rules:
      Bundle the complete set of HL7 defined tables (0001–0945) as a JSON
      file in the project. For each IS/ID field, look up the value in the
      appropriate table. Unknown values: WARNING W001.
      Tables include:
        0001: Administrative Sex (F, M, O, U, A, N, C)
        0004: Patient Class (E, I, O, P, R, B, C, N, U)
        0076: Message Type (all HL7 message types)
        0003: Event Type codes
        0005: Race codes
        0069: Hospital Service codes
        ... all 945 tables

    CATEGORY D — Segment Sequence Rules:
      After the PEG parser validates structure, re-check optional rules:
      e.g. PV2 can only appear after PV1
           OBX can only appear after OBR in ORU messages
      Cross-segment field consistency:
        PID.7 (DOB) must be before MSH.7 (message timestamp)
        PV1.44 (admit date) must be before PV1.45 (discharge date) if both set

    CATEGORY E — Encoding and Length Rules:
      Field length check against HL7 spec max lengths
      Special character encoding check (| ^ ~ \ & must be escaped)
      Null value representation check ('' vs two double quotes)

    Rule execution:
      Apply all applicable rules to each segment/field
      Collect results: { passed: Rule[], failed: { rule, segment, field, value } [] }
      Compute score: (passed.length / (passed.length + failed.length)) * 100

HL7 GENERATOR (GENERATOR_PROC):
  Input: JSON parameter object from the form UI
  Algorithm:
    Step 1: Load the appropriate template for the message type
    Step 2: A template is a JavaScript/Python object defining segment structure:
      Template = {
        segments: [
          { name: 'MSH', fields: [
            { index: 3, source: 'params.sendingApp',     default: 'HIS' },
            { index: 4, source: 'params.sendingFacility', default: 'FACILITY' },
            { index: 7, source: 'datetime.now()',         format: 'DTM' },
            { index: 9, source: 'STATIC:ADT^A01^ADT_A01' },
            { index: 10, source: 'uuid.short()' },
            { index: 12, source: 'params.version',       default: '2.5.1' },
          ]},
          { name: 'EVN', fields: [
            { index: 2, source: 'datetime.now()', format: 'DTM' },
          ]},
          { name: 'PID', fields: [
            { index: 3,  source: 'params.mrn + "^^^FACILITY^MR"' },
            { index: 5,  source: 'params.lastName+"^"+params.firstName' },
            { index: 7,  source: 'params.dob', format: 'DT' },
            { index: 8,  source: 'params.gender' },
          ]},
          ...
        ]
      }
    Step 3: Walk the template, resolve each field's source expression,
            apply formatting, handle empty/optional fields (leave empty vs omit)
    Step 4: Join fields with FIELD_SEPARATOR, join segments with \r
    Step 5: Run the Lexer + Parser on the result to validate before returning
    Step 6: If validation fails: re-examine template, apply field corrections
    Output: Valid HL7 pipe-delimited string

HL7 DETERMINISTIC AUTO-FIX:
  Input: raw HL7 message + list of validation errors from rule engine
  Algorithm:
    For each error, apply a fix function:
      E_DATE_FORMAT      : parse the date value, reformat to YYYYMMDD or YYYYMMDDHHMMSS
      E_REQUIRED_MISSING : insert a placeholder value appropriate for the data type
      E_INVALID_TABLE    : look up closest valid table value by string similarity
      E_LENGTH_EXCEEDED  : truncate the value to spec max length
      E_MISSING_SEGMENT  : insert the missing required segment with default values
      E_DELIMITER_ESCAPE : re-escape any unescaped delimiter characters in values
    Fix functions that are too risky (ambiguous data change): mark as
    MANUAL_REVIEW and do not auto-apply
    Output: corrected HL7 string + list of changes made

FHIR BRIDGE IN ALGO MODE:
  No AI call. Use a hard-coded transformation map:
    ADT PID segment → FHIR Patient resource (field-by-field mapping)
    ADT PV1 segment → FHIR Encounter resource
    ORU OBX segment → FHIR Observation resource
    ORU OBR segment → FHIR DiagnosticReport resource
    ORM OBR segment → FHIR ServiceRequest resource
  Each mapping is a list of rules:
    { hl7_segment: 'PID', hl7_field: 5, fhir_resource: 'Patient',
      fhir_path: 'name[0].family', transform: 'split("^")[0]' }
  Bundle the mapped resources into a FHIR R4 Bundle JSON structure.

BACKEND ABSTRACTION (mirror of AI mode):
  Same LLMProvider interface concept, but for the algorithm engine:
    class AlgoEngine:
      def generate(params: dict) -> str
      def parse(raw_hl7: str) -> AST
      def validate(ast: AST, profile: str) -> ValidationResult
      def autofix(raw_hl7: str, errors: list) -> FixResult
  All routing code checks engineMode and calls either LLMProvider
  or AlgoEngine — never both.

================================================================================
  SECTION 8 // REST API + WEBSOCKET SPECIFICATION
================================================================================

BASE URL: http://localhost:8000/api/v1

ALL ENDPOINTS ACCEPT engine_mode HEADER OR QUERY PARAM:
  X-Engine-Mode: cloud_ai | local_ai | algorithm
  This tells the backend which processing path to use.
  Frontend always sends the current engine mode with every request.

POST /api/v1/generate
  Headers  : X-Engine-Mode
  Body     : { mode: "form|natural_language", version, message_type,
               trigger_event, natural_language_input, params: {} }
  Response : { hl7_message, message_id, confidence, engine_used,
               generated_at, ai_reasoning?, template_used? }

POST /api/v1/parse
  Headers  : X-Engine-Mode
  Body     : { hl7_message }
  Response : { message_id, message_type, version, segments:[...],
               human_readable:{...}, fhir_bundle:{...},
               engine_used, parse_time_ms }

POST /api/v1/validate
  Headers  : X-Engine-Mode
  Body     : { hl7_message, profile }
  Response : { message_id, overall_status, overall_score,
               agents|processors: { syntax:{...}, semantic:{...}, compliance:{...} },
               total_errors, total_warnings, autofix_available,
               engine_used, validation_time_ms }

POST /api/v1/autofix
  Headers  : X-Engine-Mode
  Body     : { hl7_message, validation_errors:[...] }
  Response : { corrected_message, fixes_applied:[...],
               unfixable_errors:[...], engine_used }

POST /api/v1/transform/fhir
  Headers  : X-Engine-Mode
  Body     : { hl7_message }
  Response : { fhir_bundle:{...}, resources:[...], engine_used }

POST /api/v1/batch
  Headers  : X-Engine-Mode
  Body     : { messages:[...], profile, batch_size, stop_on_error }
  Response : streamed via WebSocket (progress events), final summary on complete

GET  /api/v1/schemas
  Response : list of available conformance profiles

GET  /api/v1/engine/status
  Response : {
    cloud_ai: { available: bool, api_key_configured: bool },
    local_ai:  { available: bool, ollama_running: bool, models: [...] },
    algorithm: { available: true, rules_loaded: 1240, version: "2.1" }
  }

GET  /api/v1/audit?from=&to=&limit=100
  Response : paginated audit log

WebSocket /ws/eventbus
  Server pushes JSON event objects:
  {
    "type": "EventType.VALIDATION_ERROR",
    "timestamp": "HH:MM:SS.mmm",
    "engine": "cloud_ai|local_ai|algorithm",
    "agent|processor": "SYNTAX_AGENT|VALIDATOR_PROC",
    "detail": "segment OBX[2] field 3 — required field missing",
    "severity": "ERROR|WARNING|INFO"
  }

Store shape:
  engineMode        : 'cloud_ai' | 'local_ai' | 'algorithm'

    lexer     : { status, metrics },
    parser    : { status, metrics },
    validator : { status, metrics },
    generator : { status, metrics }
  }
  eventBus          : EventEntry[] (last 200 events)
  sessionClock      : number (seconds since session start)
  ollamaStatus      : { running, models, activeModel }
  engineApiStatus   : { cloud_ai, local_ai, algorithm }
  batchJob          : { active, progress, total, errors, results }
  auditLog          : AuditEntry[]

Actions:
  setEngineMode(mode)     — switches engine, resets agent/processor states,
                            triggers engine health check, updates header badge
  setActiveTab(tab)       — also enforces NL_INPUT disabled in algo mode
  setCurrentMessage(msg)  — updates raw message display
  receiveWsEvent(event)   — appends to eventBus, updates relevant agent/proc
  runValidation()         — dispatches to correct backend path per engineMode
  runGenerate(params)     — dispatches to correct backend path per engineMode
